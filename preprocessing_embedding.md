o 좌표, Coordinate : 머신은 각 글자가 문서의 어디에 위치하는지를 숫자(x, y)로 파악

• **Fixed-Size Chunking** (or Token Chunking) :
o 텍스트 전체를 미리 정해진 크기(토큰 개수 또는 문자수)로 분할
• 장점 : 구현이 쉽고 빠름, 단순한 문서 처리에 적합
• 단점 : 의미적 경계(문단, 문장 등)를 무시할 수 있음, 정보 손실이나 부정확성 발생
o Chunk Overlap : 청크 끝부분 일부를 다음 청크 시작 부분에 포함
• **Recursive Chunking** :
o 문서를 여러 수준의 구분자를 가진 계층적 방식으로 나누는 기법
o 처음에는 큰 단위(문단, 단락)으로 나누고, 만약 그 청크가 설정한 최대 크기를 넘는다면
더 작은 단위(문장, 단어 등)로 반복하여 나누는 방식
• 장점 : 문서 구조를 어느 정도 보존하면서 나눌 수 있음 . 의미 단위가 파괴될 가능성 적음
• 단점 : 전체 처리 시간과 storage 요구량 증가
• **Semantic Chunking** :
o 텍스트를 의미(Semantic) 단위에 따라 분할
o 단순히 길이(토큰수, 문장수)가 아니라, 내용적 관련성이나 주제 전환을 기준으로 나뉨
• 장점 : 문맥 보존, 검색 효율성↑, 정확성 향상
• 단점 : 비용 증가, 불확실성(의미 구분이 항상 정확하지 않고, 경계가 잘못 잡힐 수 있음)

• **Hierarchical Chunking** :
o 문서에서 상위 청크(섹션)와 하위 청크(문단)를 동시에 생성하고, 검색 시 계층적으로 활용
• 장점 : 문맥 유지 + 세밀한 검색, 복잡 문서 구조에 효과적
• 단점 : **Storage**/**계산량 증가**, 어느 수준에서 분할할지 튜닝 필요
o RAG의 정밀도와 재현율을 동시에 확보하고 싶을 때
